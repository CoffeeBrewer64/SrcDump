ECard Specification Ver. 2.1

 

This document (http://intwww/bb/rf/doc/BB_Servers/ECard_v2.htm) expands the document ECard_v1 (http://intwww/bb/rf/doc/BB_Servers/ECard_v1.htm).  The version 1 document assumes the Ecard number are to entered by the customer, therefore the number must be as short as reasonable.   This version assumes the Ecard number to be entered via a barcode reader.   The length of barcode is limited by the number of digits that can be practically printed on a credit-card-sized card.

 

 

Security Properties

 

  • The Depot does not disclose E-card numbers to any other system.   (Depot does not need to trust XS) 
  • The XS has ability to verify the E-card number (XS does not need to trust Depot, and the Ecard distribution path from BroadOn to the Depot).
  • The XS cannot generate E-card numbers.  The XS does not have accesses to the E-card numbers.  (BroadOn does not need to trust iQue for E-card generation, distribution, or the protection of Ecard numbers).

 

Ecard Number

 

The Ecard number starts with a card type number.   If the first digit is 0 or 1, the card type number, T,  is 6 digits.  The card-type specifies the format and interpretation of the rest of the number.   The code range “000000” to “099999” is reserved for the BB Player Ecards.   Currently “000100” represents the bronze card, “000200” the silver card, and “000300” the gold card.  The code range “100000” to “199999” is reserved for the BB2 project.  The card-type number is followed by 2 additional numbers:  S and R.   S is a 10-digit unique serial number.   R is a 10-digit random number.   S is used to index into the database.  T is used to determine the monetary value of the card.   R is used to prevent forgery.

Barcode

 

The barcode will be encoded using Code 128C.   It is most compact code available.  Each character is encoded using 11 elements.  Each character, holding values between 0 and 102, represents two decimal digits.  The recommended barcode X-dimensional (the element width) is 8 mil or 10 mil (1 mil is 1/1000 of an inch). At 8 mil, the size of the Ecard number barcode is 1.42 x 0.25 inches.   At 10 mil, the size is 1.78 x 0.27 inches.

 

 

Other numbers

 

Define the 128-bit hash value H of the random number.

 

  H = MD5(R)

 

Generate the Message M.

 

  M = (T | S | H)

 

Define the signature of the Ecard SM.

 

  SM = RSA_Signature ( M,  PrivKey[C])

 

where PrivKey[C] is a private key held by the BroadOn HSM used to certify Ecard numbers, and C is the certificate ID of the Key used to sign the Ecard.

 

RSA_Signature and RSA_verify are the digital signature functions defined in PKCS#11 v1.5.    The length of signed message SM equals to the length of the encryption key PrivKey[C].

 

Ecard Number Generation

 

The Ecard number is generated in the BroadOn Center.   BroadOn Center maintains a Ecard database.   For example, when the cards are generated, who prints the card, the cert id used to sign the card, which XS the card uploads, etc.

 

The BroadOn center also defines the monetary value represented by T of the Ecard.   The table should be downloaded to the Depot through CDS.

 

The serial number (starting from 1262300000) is stored in the database.   The serial number is incremented when a new number is generated.  

 

The 10 digit random number is generated by the HSM.

 

A Ecard record in the BroadOn database is comprised of  T, S, R, SM, C

 

T, S, R is sent to the card printer.

 

T, S, SM, C is uploaded to the iQue Center.

 

 

Depot Operation

 

After reading a Ecard, the depot obtains T, S, R.  

 

Compute H = MD5(R).

  

Sends T, S, H to the transaction server XS.

 

 

Upload Verfication

 

No per eticket check is performed.

 

The Ecard is uploaded using database to database synchronization.    The synchronization is protected by authentication and secured by VPN. 

 

Purchase Verification

 

XS receivies T, S, and H.

 

Compute M = T | S | H.

 

Compute Result = RSA_verify(M, SM, PubKey[C] ).

If  Result is true, then the Ecard number is validated.

 

RSA Key Length and Lifetime

 

To save space, the key length is 1024 bit.  

 

Each key will be used to generate at most 4 million keys (i.e., one month generation at busy time) or 1 year.

 

The signature key generation is automatically handled by the Ecard Generation Software.    However, the software might require insertion of a higher privileged HSM token to perform signature-key generation.

 

Database Requirements

 

Keep at least 20 million Ecard in the database.

 

20 million cards added annually.  Ability to prune/compact the database.

 

Determine database transaction time and throughput on Ecard accesses.

 

Determine size requirement for 20 million Ecards.
