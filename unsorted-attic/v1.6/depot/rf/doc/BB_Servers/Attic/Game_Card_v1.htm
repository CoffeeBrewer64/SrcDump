What’s Game Card?

Game Card (similar to Phone card) is a piece of paper or plastic with a concealed random number, called the Game Card Code.  The Game Cards are physical goods sold by the retailer, and distributed using the BB Player's distribution channel.   The Game Card is a token to exchange one game.  It is single-use, i.e.,  it has no value after the first use and it cannot be recharged.    Currently Game Card is limited to game purchase, although future extension is possible,  it is beyond the scope of this document.

 

Game Card comes in a set of denominations that equal to the retail prices of the BB games.   The current proposal is to have three denominations:   “gold”, “silver”, and “bronze”.     The actual number and values of the denominations can be changed without affecting this design. 

 

People who sell Game Cards must also run a BB Depot.

 

Use case for the kids

  1. the kid buys a Game card
  2. the kid reveals the Code on the card
  3. the kid goes to the download station and selects a game
  4. the kid enters the Game Card Code
  5. XS verifies the Code and authorizes an e-Ticket

Game Card Generation and Registration Procedure

  1. IQue owns a CA in US for Game Card generation.  Its public key is Kc. Its private key is Kd.
  2. Generate a random Game Card Code, R, in US
  3. Apply one-way hash function F, on R, producing H
  4. Create certificate information, I.  It contains the dollar amount, validity period, a batch number, etc.
  5. Digitally sign the (H,I) with Kd, producing S.
  6. Register (H,I,S) to XS
  7. Ship R to the PRINTER for printing, over a secure channel

Game Card Code Verification Procedure

  1. XS reads (BB-id,R) from Depot over an encrypted channel.   
  2. XS applies one-way hash function F on R, producing H
  3. XS uses H to find the database record
  4. XS verifies I is still valid
  5. XS verifies (H,I) has valid signature S using Kc

Security Properties and Attacks

Prop 1. Nobody can recover R from (H,I,S)
Direct consequence of property of one-way hash function F.

Prop 2. Nobody can compute S from combinations of R,H,I without Kd.
Direct consequence of RSA signature algorithm, given that Kd is kept secret in US.


For the operators in XS:

Prop 3. They cannot recover the random code to print "valid" code by reading the XS database - from Prop 1.

Prop 4. They cannot insert unauthorized random code into the XS database - from Prop 2.

 

 

Attack 1.  The operators at PRINTER can print extra Game Card  using valid Game Card code.  This attack will be detected by complaints from retailer about invalid cards.

Attack 2.  The operators at XS resets the (H,I,S) record, after the code is used .  This attack should be detected from the audit record when a Code is used multiple times.

 

Length of Game Card Code

  • 48 bit random number,  generated using true (i.e., hardware-base) random number generator.
  • Converted to 16 digits of 0-9. 
  • 15 digits and 1 check digit
  • the chance of guessing one correctly is 1 in 3.3 million, assuming 10 million Code in circulation.

 

Database Management

Each Game Card has an entry in the database, indexed by hash value of the Game Card Code.   The generation procedure needs to make sure no code is reused.

 

To limit the size of database,

 

  • Used Game Card Code are backup and deleted from the database.
  • Introduce and enforce expiration date on Game cards.

 

To allow management of Game Cards for activations/deactivations,  each Game Card has a management number (also called batch number).    The management number is part of the certificate of the Game Card.

Game Card Distribution

Game cards are bundled into packs.  The minimum order unit is a pack.  A pack contains N game cards, where N is to be decided.   Each pack has a batch number.  The batch number is used to activate/deactivate of the pack of cards in the XS database.

The cards are default activiated.  No "acknowledgement" from retailer or delivery person is necessary.   In the event of robbery or lost goods, the batch numbers must be identified and the corresponding packs deactivated.  

OR

The cards are default not activated.    They are activated after confirmation of delivery.   There are 4 choice of confirmations, see below.

Expiration Date of the Game card

 

 

 

The objective is to keep the database size small. 

Card never expire!

OR

The expiration date is controlled by the digitally signed certificate.  The expiration printed on the card and the expiration date in the database can be different.  The card expires M months after the expiration date printed on the card.  For example, M=3.

What to do with expired cards  

 

 

We refund the retailer the value of expired cards.

OR

No refund.  We expect the retailer should sell the card before it expires (e.g., deep discount)

 

 

 

How to Input Game Card Code

 

Use BB Player Joystick.   Restrict digits to 1-9.  The 9 positions of the joystick selects a digit.  Press the yellow key to enter.  Another yellow key to backspace.

 

OR

 

Use a magnetic stripe reader.   Game Card is printed on a plastic card with magnetic strip.

 

OR

 

Use an optical barcode reader.

What comprises the Download Station?

 

John and Bill raised the question whether the download station should be based on BB Player

This is the existing design.  The BB Depot has 3 download stations.  Each station is a BB Player + TV, connected to the PC via USB.   Input device is the joystick.

 

OR

 

The BB Depot has 1 download station, and the station does not use BB Player.   The input and output devices are part of the PC.  A BB Player can be loosely attached to Download station to run demos.

 

OR

The BB Depot has 1 download station, and a demo station.   The download station can be a BB Player or a LCD/Touchscreen.

What are the input/output devices of the “new” download station?
